# Vocal Separation Model Evaluation

**Goal.** Evaluate alternatives to the current `htdemucs`/demucs vocal separation for the transcription pipeline (separation → crepe pitch → MIDI), using `python-audio-separator`. Two candidates were tested against the baseline: `UVR-MDX-NET-Voc_FT` (MDX, ~8s/clip — faster than htdemucs at ~12s) and `vocals_mel_band_roformer` (best raw separation quality/SDR, but ~19–21s). Each candidate lives on an isolated experiment branch (`exp/sep-mdx-voc-ft`, `exp/sep-roformer`) that is config-driven with its own DB + output dir and a shared model cache, so runs don't intermingle with `main`.

**Accuracy method & key finding.** Pipeline MIDI was scored against Hooktheory ground-truth melodies (scale-degree JSON) via a tempo-free fitting alignment with a 12-way transposition search, reported as a balanced F1 (penalizes spurious notes). The headline result: **the separation model barely affects end-to-end accuracy** (all three within ~1 F1 point). The real bottleneck is the downstream segmentation stage, which over-segments sustained notes. Re-tuning the segmentation constants (`MERGE_GAP` 0.10→0.16, `HOLD_FRAMES` 6→8, `MIN_NOTE_DURATION` 0.12→0.14) lifted overall F1 from **70.8 → 74.2 (+3.4)** — a far larger gain than any model swap. Post-tuning the Roformer edges ahead (74.7) but the margin over htdemucs (74.2) and MDX (73.6) is negligible.

**Status & next steps.** The tuned parameters were validated only at runtime (monkeypatched in the throwaway harness) and are **not committed** — all branches still carry the original defaults. The tuning was done on just 3 songs with 5 parameters, so overfitting is a real risk, evidenced by the fast/repetitive "sign-of-the-times" sample *regressing* under the merge-heavy params (no single global set suits both sustained and rapid melodies). Recommended next step (chosen direction "b"): reduce overfitting via more ground-truth songs and leave-one-song-out cross-validation before baking any constants in. Given accuracy parity, `UVR-MDX-NET-Voc_FT` remains the pragmatic model choice on speed.
